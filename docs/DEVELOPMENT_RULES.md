# 开发规则与行为准则

## 📋 核心开发规则

### 规则 1: 文档管理原则
**不要随便创建没必要的文档**

- 同类事情尽量在现有的同类文档中完成记录
- 避免文档冗余和分散维护
- 新文档仅用于：
  - 新模块的需求说明
  - 较大阶段性总结
  - 独立的配置文件

#### 📚 现有文档清单与用途

| 文档位置 | 文档名 | 用途 | 更新频率 | 用户 |
|---------|--------|------|---------|------|
| 根目录 | README.md | 项目概览、技术栈、安装指南 | 每个大版本 | 所有人 |
| 根目录 | QUICK_START.md | 3 分钟快速启动（uv/pip/Docker） | 有安装步骤变更时 | 新用户/开发者 |
| docs/ | DEVELOPMENT_PLAN.md | Phase 进度、任务清单、优先级 | 每个 Phase 完成时 | 项目规划/开发者 |
| docs/ | SYSTEM_ARCHITECTURE.md | 系统设计、模块关系、数据流、安全架构 | 新模块时更新 | 架构师/开发者 |
| docs/ | DEVELOPMENT_RULES.md | 开发规则和工作流 | 规则变更时 | 开发团队 |
| docs/ | SAFETY_SANDBOX_GUIDE.md | 沙盒执行、代码安全审查实现细节 | 需要时 | 后端开发者 |

**使用指南**：
- ❌ 不要为 Phase 完成创建新文档 → 在 DEVELOPMENT_PLAN.md 中记录
- ❌ 不要为新模块创建单独的计划文档 → 在 DEVELOPMENT_PLAN.md 对应 Phase 中补充
- ✅ 为新模块创建 API 设计/实现指南文档 → 在 docs/ 中创建（如 SAFETY_SANDBOX_GUIDE.md）
- ✅ 跨越多个 Phase 的通用指南 → 在 docs/ 中创建

---

### 规则 2: 进度跟踪集中化
**记住开发进度和系统架构更新的主要位置**

| 文档 | 用途 | 更新频率 |
|------|------|---------|
| **docs/DEVELOPMENT_PLAN.md** | Phase 进度、任务、优先级 | 每个 Phase 完成时 |
| **docs/SYSTEM_ARCHITECTURE.md** | 系统设计、模块关系、数据流 | 引入新模块时 |
| 其他文档 | 特定的配置、指南、参考 | 按需 |

**执行方式**：
1. 开发前检查这两个文档
2. 创建新文档前确认没有适合的已有文档
3. 更新完成后统一在这两个文档中体现
4. 避免在开发过程中引入过多碎片文档

---

### 规则 3: Git 提交确认制
**每次 git 提交前要跟用户确认，不能直接执行**

- 提交前先展示 diff 或变更说明
- 等待用户明确同意
- 不能和一大堆别的命令一起就执行了
- 提交信息格式：`[MODULE] Description`

**示例流程**：
1. 完成代码编辑
2. 运行测试验证
3. **展示变更内容**
4. 等待用户确认
5. 执行 `git commit`

---

## 🔄 标准工作流

```
每次开发循环：
1. 从 DEVELOPMENT_PLAN.md 开始
2. 查看当前 Phase 的任务列表
3. 查看 SYSTEM_ARCHITECTURE.md 了解系统设计
4. 执行代码编辑和测试
5. 在现有文档（通常是 DEVELOPMENT_PLAN.md）中更新进度
6. 提交 diff + 等待确认 → git commit
```

## ✅ 自检清单

开发完成前检查：
- [ ] 是否创建了不必要的新文档？
- [ ] 是否在 DEVELOPMENT_PLAN.md 中更新了进度？
- [ ] 是否在 SYSTEM_ARCHITECTURE.md 中更新了系统设计（如有新模块）？
- [ ] 是否在 README.md 中更新了项目概览？
- [ ] 是否在 QUICK_START.md 中更新了快速启动说明？
- [ ] 代码是否通过了测试？
- [ ] 是否已经展示了 git 提交内容等待确认？
